# sync-arxiv

## 1. 目标

`sync-arxiv` 的目标是把指定 arXiv 范围内的论文带入系统，并维护“论文在哪个分类、哪个归档月份里出现过”的记录。

它解决的问题是：

- 给系统建立论文主表
- 让后续 `find-repos` 有可处理的论文集合
- 让按分类、按月份、按日期范围的查询具备稳定数据基础

它不负责：

- 猜测代码仓库
- 刷新 GitHub 仓库元数据

## 2. 输入与范围

### 2.1 必填输入

- `categories`

### 2.2 支持的时间模式

- 不带时间窗口
- `day`
- `month`
- `from` + `to`

### 2.3 额外输入

- `max_results`
- `force`

其中 `max_results` 只对“不带时间窗口”的模式有意义。

## 3. 业务模式

### 3.1 无时间窗口：最新论文引导式同步

当没有 `day` / `month` / `from` / `to` 时，系统按分类抓取最新一批论文：

- 每个分类抓取一页 arXiv 搜索结果
- 默认上限是 100，允许用 `max_results` 调整
- 适合初始化、快速引导或手工 smoke test

这不是历史补全模式，只是“把最近一批论文先放进系统”。

### 3.2 单日模式

当输入是 `day` 时，系统按日期同步某一日论文。

当前实现分成两种抓取路径：

- 对“过去且距离今天不超过 90 天”的日期，优先走 arXiv catchup 页面
- 对今天，或更久以前的单日，走 submitted-date feed

这样做的目标是兼顾近期页面可读性和较老日期的可获得性。

### 3.3 整月模式

当输入是 `month` 时，系统按该月的 arXiv listing 页面抓取论文 ID，再按 ID 批量回填完整 metadata。

这是一种“先拿目录，再补详情”的模式。

### 3.4 日期范围模式

当输入是 `from/to` 时，系统按自然月切成多个执行单元。

当前行为有一个重要约束：

- 调度和 TTL 判断按请求范围工作
- 但实际抓取载体仍然是“整月 listing”

也就是说，跨月范围更像是“按月补齐相关月份”，而不是“精确裁剪到某几天的 listing 内容”。

## 4. 批量拆分规则

当 scope 跨越多个归档月份时，任务会先变成批量根任务，再拆成子任务。

子任务粒度是：

- 一个分类
- 一个归档月份

例如：

- `categories=cs.CV,cs.LG`
- `from=2026-03-20`
- `to=2026-04-10`

会拆成 4 个子任务：

- `cs.CV + 2026-03`
- `cs.CV + 2026-04`
- `cs.LG + 2026-03`
- `cs.LG + 2026-04`

## 5. 新鲜度与跳过规则

### 5.1 Past day TTL

系统会按 `category + day` 记录“这个自然日上次成功完成同步的时间”。

对已经结束的过去日期：

- 如果还在 TTL 内，则可跳过
- 如果超过 TTL，则重新同步

### 5.2 今天和未完成窗口永远视为可同步

只要请求范围触及今天或未来日期，系统会把它视为“仍可能变化”的窗口，因此不会因为 past-day TTL 而整体跳过。

这意味着：

- 今天的单日同步总是可重新执行
- 当前月通常也会被视为仍有必要同步

### 5.3 `force`

`force=true` 会跳过 TTL 判断，直接执行。

## 6. 抓取与落库流程

`sync-arxiv` 的核心流程如下：

1. 根据 scope 规划抓取单元。
2. 对每个抓取单元尝试获取资源锁。
3. 抓取 listing / search / catchup / submitted-day 内容。
4. 解析 arXiv ID。
5. 按批次调用 metadata feed，拿到完整论文信息。
6. upsert `Paper`。
7. 记录 `ArxivArchiveAppearance`。
8. 记录原始抓取快照。
9. 成功后更新 day 级完成时间，用于后续 TTL 判断。

## 7. 产出

一个成功的 `sync-arxiv` 会产出三类结果：

- 论文主体数据更新
- 归档出现关系更新
- 可追溯的原始抓取快照

对后续流程的直接影响是：

- `find-repos` 能看到更多论文
- 按分类、月份、日期范围的筛选更稳定

## 8. 关键配置项

- `ARXIV_API_MIN_INTERVAL`
- `ARXIV_SYNC_TTL_DAYS`
- `ARXIV_TRANSIENT_RETRY_LIMIT`
- `ARXIV_ID_BATCH_SIZE`
- `ARXIV_LIST_PAGE_SIZE`

它们分别影响：

- 请求节流速度
- 过去日期多久视为过期
- 临时错误的重试强度
- metadata 批量回填的大小
- listing / submitted-day 的分页大小

## 9. 当前写死但需要让用户知道的常量

### 9.1 `ARXIV_CATCHUP_MAX_AGE_DAYS = 90`

过去日期距离今天不超过 `90` 天时，单日同步优先走 catchup 页面；超过这个阈值则改走 submitted-date feed。

这意味着：

- “最近历史”与“更老历史”使用的是两套抓取载体
- 这个分界线当前不是配置项

### 9.2 无时间窗口时默认 `max_results = 100`

如果用户只给 `categories`，不带任何时间窗口，系统默认每个分类只抓最近 `100` 篇。

这意味着：

- 初始化导入默认不是全量
- 如果用户不显式调大，就只会得到一个较小的起始样本

### 9.3 arXiv 抓取并发当前固定为 `1`

单个 `sync-arxiv` 任务内部，对 arXiv 的 listing / metadata / catchup 请求流当前是串行的，没有暴露单独的并发配置项。

这意味着：

- `sync-arxiv` 的吞吐主要靠分页大小、批量大小和请求间隔调节
- 不能通过提高 arXiv 内部并发来直接提速
