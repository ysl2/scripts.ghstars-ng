# Papers table columns implementation plan

## Summary

Update the frontend Papers table to show paper and primary repository analysis fields while reusing the existing `/api/v1/papers` paginated loading path. Do not add a new frontend data loading path and do not depend on `/api/v1/repos?limit=10000` for the table's repository metadata.

## Implementation

- Extend `/api/v1/papers` summary rows with `journal_ref` and primary GitHub repository metadata: language, size in KB, created time, pushed time, and description.
- Reuse the existing per-page batch lookup in `/api/v1/papers`: collect primary repository URLs for the current page and query `github_repos` once for that page.
- Update frontend `PaperSummary` and row mapping to consume the extended `/papers` payload.
- Replace the Papers table columns with `Status`, `ID`, `Title`, `Category`, `Published`, `Repo`, `Stars`, `Language`, `Size`, `Created`, `Pushed`, `Journal Ref`, `Comment`, `Description`, and `Author`.
- Remove all hidden Papers column definitions, including the internal search blob column, and bump the Papers grid persistence id to avoid stale localStorage state.
- Format `Author` as the first three authors plus `et al.` when needed.
- Format repository size as `KB`, `MB`, or `GB`, with up to two decimal places and no unnecessary trailing zeroes.

## Tests

- Add backend public API coverage for the new summary fields when repository metadata exists.
- Add backend public API coverage for missing repository metadata returning `null` repository fields while preserving paper fields.
- Add backend public API coverage proving paged `/papers` results include repository metadata on later pages.
- Run `tests/test_public_api.py` and frontend `npm run build`.
