# Re-run 保持浏览上下文

## 摘要

修正 `re-run` 后 drawer 关闭、batch folder 收起、attempt history 折叠的问题。用户视角下 `re-run` 只是当前 job/attempt 进入下一次运行，不应该改变当前打开和展开的浏览上下文；代码层面即使产生新 job id，也要在 UI 内平滑 handoff，不出现先关闭再打开的闪烁。

## 关键改动

- 继续使用现有 `POST /jobs/{job_id}/rerun` 返回的 `JobRead`，不改后端 API、不改数据库。
- `rerunExistingJob` 成功后读取返回的新 job，并根据当前 UI 上下文决定是否迁移 selection。
- 如果当前 drawer 正在查看被 re-run 的 job 或同一 attempt chain，立即切到返回的新 job id，并用返回的 `JobRead` 填充 detail，避免等待列表刷新导致 drawer 闪烁。
- 如果用户从 batch folder 详情里的 child list 点击 child re-run，但 drawer 当前看的是 batch folder，则保持 batch folder 详情不变，只刷新 children。
- 如果 batch folder root 被 re-run 且旧 batch folder 已展开，把展开状态迁移到新 batch folder id；如果新 row 尚未出现在列表中，先保留展开意图，等 jobs 刷新后自然展开。
- `expandedJobGroups` 继续以 `attempt_series_key` 为 key；re-run 不移除该 key，并在 re-run 成功后强制刷新已展开 attempt history。
- `handleDisplayedKeysChange` 增加 rerun handoff 保护：如果 selected job 是刚刚 re-run 返回的新 job，暂时不要因为它尚未出现在 visible keys 中就清空 selection。
- 对 direct jobs、batch folders、batch child jobs 统一使用该行为；`export` 当前不可 re-run，不需要特殊处理。

## 状态处理

- 新增短生命周期 frontend ref/state，用来记录 re-run 返回的新 selected job id，防止列表刷新间隙关闭 drawer。
- 当新 id 已经出现在 jobs/child rows/history rows 中，或者用户主动关闭 drawer、切换 tab、选择其他 row 时，清除该 handoff 保护。
- 对 batch child re-run，如果其 parent batch 当前展开，则强制刷新该 parent 的 child cache。
- 对 batch root re-run，如果旧 parent 展开，则把 `expandedParentJobIds` 中旧 id 替换或补充为新 id，并清理/刷新新 parent 的 child cache。
- 对已展开 attempt history，如果对应 `attempt_series_key` 已展开，则保留展开状态并强制 reload history。

## 测试

- 手测 direct job re-run：drawer 始终打开，selection 直接切到新 attempt，attempt history 展开状态保持。
- 手测 batch folder re-run：drawer 始终打开，batch folder 展开状态保持，旧 id 到新 id 的切换不闪烁。
- 手测 batch child re-run：从主表 child row 和 batch drawer child list 两个入口分别验证；parent folder 不收起，attempt history 不折叠。
- 回归验证 batch child 时间倒序展示仍然生效。
- 运行 `uv run python -m pytest -q` 和 `npm --prefix frontend run build`。

## 假设

- 本次不修改 `docs/design`。
- 本次只修正 frontend re-run 后的 selection/expanded state handoff，不改变 job 执行、rerun 后端语义或排序规则。
