# 统一断点续跑模型实现计划

## 目标

统一 `sync-papers`、`find-repos`、`refresh-metadata` 的 resume 用户心智：TTL 表示 freshness，resume 表示 retry progress。实现采用两层模型：公共 item-level resume + `sync-papers` 现有 request-level checkpoint。

## 实现步骤

1. 更新最新 design snapshot，补充统一 Resume Model，并明确 `sync-papers` 保留 request-level checkpoint，`find-repos` / `refresh-metadata` 采用 item-level resume。
2. 测试先行：新增覆盖 `find-repos` repair 跳过已完成 paper、force 不复用 item resume、`refresh-metadata` repair 跳过已完成 repo、API repair summary 的测试。
3. 新增通用 item resume progress 数据模型和迁移，唯一键为 `attempt_series_key + job_type + item_kind + item_key`。
4. 接入 `find-repos`：成功持久化 paper 后记录 completed item；repair run 读取同 series completed papers 并跳过。
5. 接入 `refresh-metadata`：成功处理 repo 后记录 completed item；repair run 读取同 series completed repos 并跳过。
6. 扩展 `repair_resume_json` 和 stats：新增 `resume_items_reused`、`resume_items_completed`，并保留 `sync-papers` 的 `checkpoint_*` 字段。
7. 跑后端测试，重点回归 sync checkpoint、batch rerun、public API、sync policy。

## 边界

- 不引入 provider raw response 通用 checkpoint 表。
- 不为 `refresh-metadata` 引入 TTL。
- 不改变三条链路的 provider 顺序、限速、batch 展开规则。
- 不改变 `sync-papers` 现有 checkpoint 语义。
