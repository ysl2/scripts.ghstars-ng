# Re-run Handoff 最小修复

## 摘要

本次只修可控的前端交互问题：`re-run` 后 drawer 不应因为新 job row 暂时不在表格中而关闭，batch folder 和 attempt history 的展开状态也不应被清掉。不改后端 job 生命周期，不提前创建 batch children，不新增 API/DB。

## 关键改动

- 只改前端 handoff 逻辑，核心在 `frontend/src/App.tsx`。
- 保留现有 rerun API 返回的新 `JobRead`，成功后立即把 selected job handoff 到新 job id，并用返回值填充详情。
- 修正 `selectJob(null)` 路径：如果当前处于 rerun handoff，且这个 `null` 是 grid 因新 row 暂时不存在触发的临时取消选择，则忽略，不关闭 drawer。
- 用户主动关闭 drawer、切换 tab、点击其他 row 时，正常清除 handoff 状态。
- batch folder rerun 时继续保留展开 id 迁移逻辑，但不要求 queued 阶段立即有 children。
- attempt history 继续按 `attempt_series_key` 保持展开并刷新，不新增新的展开状态模型。

## 不做的事

- 不改 `AgGridSheet` 公共接口。
- 不改 batch root 创建 children 的时机。
- 不做 planned child preview。
- 不做“加载时保留旧 children 直到新 children 出现”的平滑 UI。
- 不改 queue、job 执行、rerun 后端语义和 batch child 排序规则。

## 测试

- 手测 batch folder drawer：打开详情后点 `Re-run`，drawer 全程不关闭，不出现先关再开的闪烁。
- 手测 batch folder 展开：点 `Re-run` 后最外层展开状态保持；queued 阶段 children 可能为空，这是本次接受的现状。
- 手测 child/direct job：点 `Re-run` 后 drawer 切到新 attempt，attempt history 不折叠。
- 运行 `npm --prefix frontend run build`。
- 运行 `uv run python -m pytest -q`。

## 假设

- 本次不修改 `docs/design`。
- 现有 queued batch folder children 为空是后端生命周期行为，本次明确不处理。
- 本次目标是消除 drawer 关闭和展开状态丢失，不追求 children 列表在 queued 阶段完整展示。
