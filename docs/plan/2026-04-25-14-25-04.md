# Enforce serial queue after worker restart

## Context

After `docker compose up -d --build`, the old worker container can exit while its current job remains marked `running` in the database. The new worker currently claims the next pending job even when a fresh `running` job already exists, so the UI can show two running child jobs under the same batch folder.

This is a queue claim bug. It is not caused by the frontend chevron CSS change.

## Plan

1. Add a claim-time guard that checks for any non-stale `running` job before claiming new work.
2. If a non-stale `running` job exists, return no claim so the single worker idles until the existing lock becomes stale or the job finishes.
3. Keep existing stale-running recovery behavior unchanged, so stale running jobs can still be reclaimed after `JOB_QUEUE_RUNNING_TIMEOUT_SECONDS`.
4. Add tests for both cases:
   - a fresh running job blocks claiming a pending job;
   - a stale running job can still be reclaimed.
5. Run the focused job tests, then the full backend test suite if the focused tests pass.

## Expected Behavior

- The serial queue should not claim a second job while any fresh running job exists.
- A worker restart may leave one visible running job until timeout, but it should not create another running job.
- Stale running jobs remain recoverable through the existing timeout mechanism.
